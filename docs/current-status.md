## What Just Happened (Session 4)

### Engine is now fully data-driven — loads any conforming GameState JSON

**Generalized level loader:**
- `loadLevel(url)` in `game-loop.js` fetches any conforming GameState JSON
- Reads tilemap, tileset, physics, entities, player spawn — all from fixture data
- All fields have sensible fallbacks — won't crash on minimal fixtures
- No hardcoded constants remain in any module

**Dynamic level dimensions:**
- Removed hardcoded `MAP_W=210`, `MAP_H=15`, `TILE=16` from `state.js`
- Replaced with `state.mapW`, `state.mapH`, `state.tileSize` — set at load time from JSON
- All five modules (renderer, camera, collision, entities, physics) updated to use dynamic state

**URL parameter level selection:**
- `?level=data/test-fixtures/episode1.json` loads directly
- `?level=data/test-fixtures/episode2.json` loads directly
- No parameter → level select overlay with buttons for both fixtures
- Backtick key (`` ` ``) toggles level select (Escape avoided — browsers intercept it)

**Second test fixture — episode2.json (Iron Caves):**
- 90×15 tiles (vs 210×15 in episode1) — validates dynamic map dimensions
- Physics grounded in Mega Man 2 ROM-disassembled data (not invented):
  - Constant gravity (0.25 ascending + falling) — no two-phase split like SMW
  - run_speed: 1.5 px/frame, jump_velocity: -5.5, terminal_velocity: 4.0
  - Instant movement feel via high acceleration (0.75) + high friction (0.2)
- Dark cave palette (#0A0A2A void, gray stone)
- 4 enemies with faster patrols
- Generated by `tools/gen-episode2.js`

**Double-loop speed bug found and fixed:**
- `requestAnimationFrame(loop)` was called twice — once in init, once in level-select button handler
- Two parallel physics loops = 2× speed on everything
- Fixed: loop starts exactly once at top of init

**Geometry validation:**
- Max jump height calculated from MM2 physics (67.7px / 4.2 tiles)
- Pit #2 narrowed from 4→3 tiles; Pits #3 & #4 got bridge platforms
- Note: level 2 still has an unscalable barrier near the end — acceptable for now, this is what Gate 1 diagnostics will catch systematically

**Physics fidelity note:**
- Both fixtures use real game physics data but feel is not yet 1:1 with source games
- Acceptable for current stage — physics tuning is future diagnostic work
- The important validation: engine loads and applies different physics models from different fixtures
